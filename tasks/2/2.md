# Задание 2

Реализовать REST API для управления списком контактов с хранением данных в памяти.

## Функциональные требования

1. Для всех эндпоинтов сервис принимает и отдает данные в формате `JSON`.
   В ответе всегда проставлен заголовок
   `Content-Type: application/json`.

1. Пользователь может добавить новый контакт с помощью эндпоинта
   `POST http://127.0.0.1:8080/api/v1/contacts`.

   Пример тела запроса:

   ```json
   {
     "first_name": "Александр",
     "middle_name": "Сергеевич",
     "last_name": "Пушкин",
     "birthday": "1799-06-06",
     "description": "Великий поэт"
   }
   ```

   Пример тела ответа:

   ```json
   {
     "id": 1,
     "first_name": "Александр",
     "middle_name": "Сергеевич",
     "last_name": "Пушкин",
     "birthday": "1799-06-06",
     "description": "Великий поэт"
   }
   ```

   Полная `OpenAPI` схема объекта `Contact`, использующаяся во всех запросах и
   ответах к `api` сервиса:

   ```yaml
   components:
     schemas:
       Contact:
         type: object
         required:
           - first_name
           - last_name
           - birthday
         properties:
           id:
             type: integer
             format: int64
             readOnly: true
           first_name:
             type: string
             minLength: 1
             maxLength: 255
           middle_name:
             type: string
             minLength: 1
             maxLength: 255
           last_name:
             type: string
             minLength: 1
             maxLength: 255
           birthday:
             type: string
             format: date
           description:
             type: string
             minLength: 1
             maxLength: 255
   ```

1. Пользователь может получить список всех контактов с помощью
   эндпоинта `GET http://127.0.0.1:8080/api/v1/contacts`.

1. Пользователь может получить информацию о конкретном контакте с помощью
   эндпоинта `GET http://127.0.0.1:8080/api/v1/contacts/{id}`.

1. Пользователь может удалить контакт с помощью
   эндпоинта `DELETE http://127.0.0.1:8080/api/v1/contacts/{id}`.

1. Сервис должен возвращать ошибки с кодом `400` в случае
   невалидного запроса на создание контакта и `404` в случае попытки
   обратиться к несуществующему `ID` (получить или удалить контакт).
   Возвращаемая ошибка должна быть в формате `JSON` и содержать причину.  
   Допустимые значения для объекта `Contact` указаны в `OpenAPI` схеме выше.

   ```yaml
   components:
     schemas:
       ValidationError:
         type: object
         required:
           - field
           - reason
         properties:
           field:
             type: string
           reason:
             type: string
       Errors:
         type: object
         required:
           - errors
         properties:
           errors:
             type: array
             items:
               $ref: '#/components/schemas/ValidationError'
   ```

## Нефункциональные требования

1. Уникальный `ID` контакта генерируется на стороне сервиса. Выбор алгоритма
   генерации остается на усмотрение разработчика.

1. Контакты хранятся в оперативной памяти сервиса. Потеря списка контактов при
   перезапуске сервиса считается допустимой.

1. Для валидации запроса можно использовать любую удобную библиотеку или
   написать свою.

## Ограничения

1. Для построения `api` допускается использовать только библиотеку `net/http`
   и предоставляемый ею роутер.

## Об инициализации объектов в Go

Многие разработчики, пришедшие в `Go` из таких языков как `C++`, `Java` или
`Python`, часто пишут код создания и инициализации объекта следующим образом:

```go
type Point struct {
    x, y int
}

func New() *Point {
    return &Point{}
}

func (p *Point) Init(x, y int) {
    p.x = x
    p.y = y
}
```

В `Go` применение `New` в паре с `Init` считается антипаттерном, так как
усложняет код и разбивает процесс создания объекта на две части. Идиоматичный
вариант:

```go
type Point struct {
    x, y int
}

func New(x, y int) *Point {
    return &Point{x: x, y: y}
}
```
